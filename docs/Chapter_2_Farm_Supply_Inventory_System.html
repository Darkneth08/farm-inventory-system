Chapter 2: Design Artifacts and Analysis

Farm Supply Inventory System

VI. Wireframes

The actual Farm Supply Inventory System uses a shared application shell with a left sidebar, a top header, and role-based views. After login, the first screen depends on the user role: customer users land on Customer Home, staff users land on Staff Desk, and manager, admin, and super_admin users land on Dashboard. Because the PDF requires a dashboard wireframe, the first diagram below follows the real Dashboard page used by manager, admin, and super admin users in the current system.

  1. The POS page begins with a dedicated POS Workspace hero and step strip, not with a plain form.
  2. The warehouse selector is required because the catalog is warehouse-aware and loads live stock from the chosen source.

  4. The payment area changes dynamically based on the selected payment method and discount type.
  5. Cash mode shows Amount Tendered and quick-cash buttons, while GCash mode shows reference and receipt upload fields.

  8. After a successful sale, the result appears in the Sale result card and can be printed through Print Last Receipt.

VII. Task Analysis

Main transaction analyzed: Staff POS checkout

Step User Action System Response
1 A staff user opens the system and logs in. The system authenticates the account and opens Staff Desk as the default staff landing view.
2 The user clicks Open POS from Staff Desk, or opens POS Counter from the sidebar. The system switches to the Point of Sale view.
3 The user selects a warehouse. The system loads the live POS catalog, warehouse label, stock-source note, and POS header metrics.
4 The user searches the catalog and enters item quantities. The system filters product cards and shows availability, reorder point, batch count, and expiry information.
5 The user clicks Add to Sale on one or more products. The system adds products to the sale cart and updates items, subtotal, discount, grand total, and payment status.
6 The user enters optional customer name, selects payment method, and chooses a discount type. The system reveals the correct conditional fields such as cash tendered, GCash reference and receipt upload, or senior/PWD verification inputs.
7 The user reviews the cart and adjusts quantities if needed. The system recalculates subtotal, discount, amount tendered, balance due, and change in real time.
8 The user clicks Complete Sale. Laravel validation checks request fields, required proof inputs, payment readiness, and stock availability in the selected warehouse.
9 If the transaction is valid, the system stores the sale. The system creates sale records, sale items, related stock-out transactions, and deducts inventory from the correct warehouse batches.
10 The transaction finishes successfully. The system shows the success summary in Sale result, adds the sale to Recent POS Sales, and enables Print Last Receipt.
